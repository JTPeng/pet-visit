# 宠物上门喂养小程序 - 产品需求与设计文档

> 创建日期：2026-05-09
> 文档版本：v1.0（MVP 设计）
> 产品代号：pet-visit

---

## 一、产品定位

### 1.1 核心定位
**D 类：社区 + 服务混合型宠物小程序**

先以社区 + 工具沉淀宠主用户，验证留存与付费意愿后，再向上门喂养服务延伸；长期演进至 A 类平台撮合模式。

### 1.2 差异化定位
- 垂直于**宠物日常记录与养宠社区**
- 以宠物档案作为账户黏性核心
- 以养宠内容作为流量入口
- 上门喂养等服务作为后期变现延伸

### 1.3 不做什么（MVP 阶段明确边界）
- 不做自营上门喂养服务
- 不做撮合交易（不碰资金结算）
- 不做短视频、直播
- 不做私信、同城
- 不做用户实名认证

## 二、目标用户

| 用户类型 | 特征 | 需求 |
|---|---|---|
| **核心用户** | 城市年轻养宠人群（25-40 岁），养 1-3 只猫/狗 | 记录宠物成长、疫苗驱虫提醒、交流养宠经验 |
| **潜在用户** | 初次养宠的新手 | 学习养宠知识、寻求经验帮助 |
| **延伸用户（后期）** | 有上门喂养/寄养/遛狗需求的宠主 | 寻找可信赖的喂养服务 |

## 四、技术架构

### 4.1 技术栈

| 层级 | 选型 | 说明 |
|---|---|---|
| 前端 | **uni-app + Vue 3 + TypeScript** | 已初始化，多端能力保留 |
| 前端主投放平台 | **微信小程序** | MVP 先只投放微信端 |
| 后端 | **NestJS（Node.js）** | 模块化框架，与 MVP 模块化原则契合 |
| 数据库 | **PostgreSQL + Prisma ORM** | 关系型，类型安全，后期扩展友好 |
| 对象存储 | **腾讯云 COS / 阿里云 OSS** | 小程序端直传 + CDN |
| 部署 | **单台轻量云服务器 + Docker** | 起步方案，MVP 足够 |
| 缓存 | 暂不引入 Redis | 流量起来后再加 |

### 4.2 模块化架构原则

**所有代码必须按模块组织，确保可扩展性。**

后端目录结构（NestJS）：

```
src/
├── modules/
│   ├── auth/           # 用户登录、手机号绑定
│   ├── user/           # 用户资料、关注关系
│   ├── pet/            # 宠物档案
│   ├── post/           # 社区帖子
│   ├── comment/        # 评论
│   ├── interaction/    # 点赞、收藏
│   ├── report/         # 举报
│   ├── moderation/     # 内容审核（对接微信接口）
│   ├── upload/         # 对象存储直传鉴权
│   └── admin/          # 后台管理（基础版）
├── common/             # 公共装饰器、拦截器、过滤器
├── config/             # 配置管理
├── prisma/             # Prisma schema & migrations
└── main.ts
```

前端目录结构（uni-app）：

```
src/
├── pages/
│   ├── index/          # 首页（社区信息流）
│   ├── post/           # 发帖相关
│   ├── detail/         # 帖子详情
│   ├── profile/        # 个人中心
│   ├── pet/            # 宠物档案
│   └── user/           # 他人主页
├── components/         # 通用组件
├── api/                # 按后端模块划分接口文件
├── stores/             # Pinia 状态管理（按领域划分）
├── utils/
└── types/              # TypeScript 类型定义
```

### 4.3 扩展位预留（重要）

虽然 MVP 只做档案 + 社区，但以下扩展位要在架构设计中预留，**避免后期大改**：

| 扩展方向 | 预留方式 |
|---|---|
| 工具集（疫苗/驱虫/体重提醒） | 宠物档案表预留 `extra_info` JSON 字段；独立 `reminder` 模块槽位 |
| AI 识宠/识病 | 独立 `ai` 模块槽位，走外部 API 封装 |
| 上门喂养预约 | 独立 `service` 模块槽位；订单状态机抽象 |
| 喂养师端 | 用户表预留 `role` 字段；用户类型枚举扩展点 |
| 支付 | `payment` 模块槽位；订单表设计时考虑分账字段 |
| 付费会员 | `membership` 模块槽位；用户表预留 `member_level` 字段 |
| CPS 分销 | `promotion` 模块槽位 |
| 广告位 | 前端广告位组件化，后端 `ad_config` 配置化 |
| 私信/消息 | `message` 模块槽位；通知中心抽象 |
| 同城 | 用户表预留 `location` 字段；PostgreSQL 可启用 PostGIS |

## 五、分阶段演进路线

### 阶段 1：MVP（当前阶段，个人主体）
- 功能：宠物档案 + 社区
- 变现：暂无（达到 1000 UV 后开流量主）
- 主体：个人
- 目标：验证留存，积累种子用户

### 阶段 2：轻度变现（个人主体 + 流量主）
- 新增：微信激励视频广告（仅宠主端）、Banner
- 新增：CPS 分销（京东联盟 / 多多进宝 / 淘宝联盟）
- 主体：个人
- 目标：覆盖服务器成本 + 小额营收

### 阶段 3：服务 + 付费（升级个体工商户）
- 升级主体：**个体工商户** + 对公账户 + 微信支付商户号
- 新增：付费会员（去广告 + AI 次数 + 高级提醒）
- 新增：付费内容（训犬课/养宠课）
- 新增：上门喂养预约（自营或合作店导流模式，不做撮合）
- 新增：工具集（疫苗/驱虫/体重提醒）
- 新增：AI 识宠/识病（看广告解锁次数）

### 阶段 4：平台化（升级企业主体）
- 升级主体：**企业** + 电商收付通 + 保险 + 法务文档
- 新增：喂养师端
- 新增：订单担保交易、分账
- 新增：喂养师实名 + 背调 + 评价体系
- 新增：到家打卡、过程录像、离开打卡
- 目标：正式转型 A 类平台撮合模式

## 八、功能模块详细需求

### 8.1 用户体系模块

#### 8.1.1 登录流程
- 用户打开小程序 → 微信静默登录 → 后端获取 openid + unionid → 返回 JWT Token
- 浏览内容不要求手机号
- 首次**发帖 / 评论 / 点赞 / 关注 / 修改资料**时弹窗引导绑定手机号

#### 8.1.2 手机号绑定
- 调用微信 `button open-type="getPhoneNumber"` 一键获取
- 后端通过微信 API 解密获取手机号
- 一个手机号可绑定多个 openid（多端场景），一个 openid 只绑定一个手机号

#### 8.1.3 个人资料
- 字段：昵称（≤20 字符）、头像（上传 OSS）、个人简介（≤100 字符）
- 昵称/头像修改走 msgSecCheck/imgSecCheck 审核
- 初始默认昵称"养宠人_随机 4 位数字"、系统默认头像

#### 8.1.4 其他用户主页
- 展示：头像、昵称、简介、宠物数量（不展示具体宠物）、发帖数、粉丝/关注数
- 展示该用户的发帖列表
- 可关注/取消关注

---

### 8.2 宠物档案模块（极简版）

#### 8.2.1 档案字段
| 字段 | 类型 | 必填 | 备注 |
|---|---|---|---|
| 宠物头像 | 图片 | 否 | 默认占位图 |
| 宠物昵称 | 字符串（≤20） | 是 | msgSecCheck 审核 |
| 品种类 | 枚举 | 是 | 猫/狗/其他 |
| 品种小类 | 字符串 | 否 | 下拉选择器 + 自定义输入 |
| 性别 | 枚举 | 是 | 公/母/未知 |
| 生日 | 日期 | 否 | 可选"约几岁" |
| 当前体重 | 数字（kg） | 否 | |
| 备注 | 字符串（≤200） | 否 | |

#### 8.2.2 管理能力
- 一个用户可添加多只宠物（无数量上限，常识限制即可）
- 可编辑、可删除
- 档案**私有**：仅本人可见
- 档案是"账户"的组成部分，不与社区强耦合

#### 8.2.3 扩展位预留
- 数据表预留 `extra_info JSONB` 字段，后续疫苗/驱虫/体重记录/饮食等扩展不需要改表结构
- 预留"公开"开关字段，为后续档案公开分享功能做准备

---

### 8.3 社区模块（轻量图文版）

#### 8.3.1 发帖
- 内容：文字（≤1000 字符）+ 图片（1-9 张，单张 ≤5MB）
- 可选：关联宠物（选择自己的一只宠物，作为发帖关联）
- 可选：添加话题标签（从预设列表选，最多 3 个）
- 发布前走 msgSecCheck + imgSecCheck 审核
- 命中可疑 → 进入人工复审队列；命中违规 → 拒绝并提示用户

#### 8.3.2 预设话题标签（初期）
- #晒猫
- #晒狗
- #养宠日常
- #求助
- #萌宠瞬间
- #疫苗驱虫
- #养宠新手

（后期可由运营后台动态管理，MVP 阶段硬编码）

#### 8.3.3 信息流
- **推荐流（首页默认）**：
  - 综合排序分数 = 热度分 × 时间衰减因子
  - 热度分 = 点赞数 × 1 + 评论数 × 2 + 收藏数 × 1.5
  - 时间衰减因子：24 小时内 = 2.0，24-72 小时 = 1.0，72 小时以上 = 0.5
  - 按综合分数倒序分页
  - 说明：MVP 阶段不做"已读去重"（避免引入曝光历史表），用户刷新时顺序可能轻微重复，可接受
- **关注流**：
  - 纯时间倒序
  - 仅展示已关注用户的发帖

#### 8.3.4 帖子详情页
- 内容：图文、发布者信息、发布时间
- 互动：点赞、评论、收藏、转发（生成小程序分享卡片）
- 评论区：二级评论（楼中楼）、按时间倒序或热度排序
- 举报按钮（三个点菜单）

#### 8.3.5 评论
- 纯文字（≤500 字符）
- 走 msgSecCheck 审核
- 支持 @用户（后期扩展，MVP 暂不做 @ 提醒）

#### 8.3.6 点赞、收藏
- 点赞：帖子、评论均可
- 收藏：仅帖子可收藏
- 收藏夹：个人中心 → 我的收藏（简单列表）

#### 8.3.7 关注关系
- 单向关注
- 在帖子详情、用户主页可关注/取消关注
- 关注流在首页可切换查看

---

### 8.4 内容审核模块

#### 8.4.1 机器审核
- **文本**：微信 `msgSecCheck`
  - 场景：发帖内容、评论内容、昵称、简介、宠物昵称、宠物备注
  - 安全级别：内容安全级别 2
- **图片**：微信 `imgSecCheck`
  - 场景：帖子图片、用户头像、宠物头像

#### 8.4.2 本地敏感词库
- 词库分类：政治、色情、赌博、医疗广告、引流外联
- 词库初期可从开源项目引入（如 GitHub `sensitive-words` 等）
- 管理后台可添加/删除
- 命中本地敏感词 → 直接拒绝，不再调用微信接口（节省调用次数）

#### 8.4.3 人工审核队列
- 微信接口返回"疑似"时进入队列
- 管理员在管理后台看到待审队列，人工通过/拒绝
- 队列默认隐藏内容，不对其他用户可见

#### 8.4.4 举报
- 每条帖子、每条评论、每个用户主页可举报
- 举报原因枚举：色情低俗 / 政治敏感 / 广告引流 / 人身攻击 / 虚假信息 / 其他
- 举报入队管理后台，达到阈值（如 3 次）自动隐藏

#### 8.4.5 违规处罚
- 违规一次：内容删除 + 警告
- 违规两次：禁言 7 天
- 违规三次：永久封号
- 封禁数据存储在用户表 `ban_until` 字段

---

### 8.5 管理后台（最小化）

MVP 阶段可先做**极简命令行/简易后台**：

- 待审核内容队列（通过/拒绝）
- 举报列表处理
- 敏感词库维护
- 用户封禁/解封

形式建议：独立 Web 页面（Next.js 或 NestJS + 简单 HTML/Vue 模板），账号密码登录，仅你本人使用。

---

### 8.6 对象存储（OSS/COS）模块

- 前端调用后端 `/upload/sign` 接口获取临时上传凭证
- 前端直传 OSS/COS
- 前端上传成功后将 URL 提交给后端落库
- 路径规则：`{env}/{module}/{yyyy-mm}/{uuid}.{ext}`
  - 例：`prod/post/2026-05/xxx.jpg`
- 图片访问走 CDN 子域名
- 原图上传，访问时按需传参压缩（OSS/COS 图片处理参数）

## 九、非功能性需求

### 9.1 性能
- 首屏加载 ≤ 2s
- 社区信息流分页（每页 20 条），下拉刷新、上拉加载更多
- 图片懒加载、CDN 加速、按需压缩

### 9.2 稳定性
- 后端接口全部加日志（请求/响应/错误）
- 关键接口增加限流（如发帖 10 秒/次、评论 3 秒/次）
- 数据库定期备份（MVP 阶段可用云数据库自动备份）

### 9.3 安全
- 所有接口走 HTTPS
- JWT Token 有效期 7 天，支持刷新
- 数据库密码、API Key 等敏感配置通过环境变量注入，不进仓库
- 防止 SQL 注入（Prisma 天然防御）
- 防止 XSS（前端渲染统一转义）
- 防止接口刷（登录态鉴权 + 限流）

### 9.4 可观测性（MVP 最小）
- 后端 winston/pino 日志落盘
- 接微信小程序官方统计
- 错误监控暂不接（用户量大了再接 Sentry 等）

### 9.5 开发规范
- TypeScript 严格模式
- ESLint + Prettier 统一代码风格
- Git commit 规范（feat/fix/chore/docs 等前缀，已在现有仓库采用）
- 所有模块独立，模块间通过依赖注入 + 接口定义通信
- 单元测试覆盖核心业务逻辑（用户、宠物、帖子 CRUD）

### 10.1 宠物档案扩展

## 十二、风险与备注

| 风险 | 应对 |
|---|---|
| 个人主体类目受限 | 选"工具/社交"类目，描述为"宠物日常记录与养宠社区"，避免"宠物服务"表述 |
| 内容审核被投诉 | 严格双审 + 举报闭环 + 封禁机制；上线初期人工抽查 |
| 内容冷启动困难 | 自产种子内容 + 小红书等外部引流 + 邀请身边宠主 |
| 广告审核不过 | 阶段 1 不接广告，等 UV 达标后再接官方流量主；禁止强制广告设计 |
| 后期升级架构成本 | MVP 即按模块化设计 + 扩展位预留，最小化改造 |
| 多端分化 | uni-app 保留多端能力，但 MVP 只专注微信小程序测试与发布 |
